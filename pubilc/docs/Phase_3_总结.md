### 2.2 技术实现要点

#### 2.2.1 向量数据库集成
- **实现方案**: ChromaDB 本地向量存储
- **核心功能**: 持久化存储、向量相似度搜索、元数据过滤
- **技术特点**: 轻量级、易于集成、支持批量操作

#### 2.2.2 Embedding 服务
- **双模型支持**: OpenAI Embedding + 本地 Ollama 模型
- **批量向量化**: 支持文档批量处理
- **配置管理**: 通过配置文件和环境变量控制

#### 2.2.3 文档处理
- **Document Washer**: 支持 Markdown、TXT、PDF 格式清洗
- **Chunker**: 实现语义分块、递归分块、固定长度分块
- **元数据增强**: 为每个块添加丰富的元数据信息

#### 2.2.4 检索系统
- **Searcher**: 基于向量相似度的 Top-K 检索
- **Reranker**: Cross-encoder 精排 + 降级简单排序
- **模型配置**: 使用 `bbjson/bge-reranker-base` 模型

#### 2.2.5 RAG Pipeline
- **完整流程**: 清洗 → 分块 → 向量化 → 存储 → 检索 → 重排
- **统一接口**: 提供文档处理和查询的标准接口
- **错误处理**: 完善的异常处理机制

#### 2.2.6 知识库功能
- **笔记导入**: 支持多种格式的笔记批量导入
- **风格学习**: 从笔记中提取写作风格特征
- **风格管理**: 风格存储、技能化、参考段落检索

## 4. 技术架构

### 4.1 模块结构

```
rag/
├── document_washer.py   # 文档清洗
├── chunker.py           # 文档分块
├── embedding.py         # 向量化服务
├── searcher.py          # 向量检索
├── reranker.py          # 结果重排序
├── pipeline.py          # RAG 处理管道
├── note_importer.py     # 笔记导入
├── style_learner.py     # 风格学习
├── style_manager.py     # 风格管理
└── vector_db_optimize.py # 向量数据库优化
```

### 4.2 核心流程

#### RAG 处理流程
```
原始文档
    ↓
Document Washer (清洗)
    ↓
Chunker (分块)
    ↓
Embedding Service (向量化)
    ↓
Vector DB (存储)

查询时:
查询文本
    ↓
Embedding (向量化)
    ↓
Searcher (向量检索 Top-K*2)
    ↓
Reranker (精排 Top-K)
    ↓
返回结果
```

## 7. 技术挑战与解决方案

### 7.1 挑战 1: 模型加载优化
**挑战**: Reranker 模型加载时间长，影响测试速度
**解决方案**: 使用 Mock 技术避免实际加载模型，确保测试快速运行

### 7.2 挑战 2: 配置管理
**挑战**: 多模型配置管理复杂
**解决方案**: 统一配置管理，支持环境变量覆盖

### 7.3 挑战 3: 分块策略选择
**挑战**: 不同文档类型需要不同的分块策略
**解决方案**: 实现多种分块策略，根据文档类型自动选择

### 7.4 挑战 4: 元数据管理
**挑战**: 确保元数据在整个 RAG 流程中正确传递
**解决方案**: 设计统一的元数据传递机制，确保信息完整
