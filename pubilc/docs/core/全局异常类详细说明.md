# 全局异常类详细说明

## 1. 异常体系概述

ELEVEN Blog Tuner 采用了分层、结构化的异常处理体系，提供了从通用到具体的异常层级结构，确保错误处理的一致性和可追踪性。

### 1.1 设计理念

- **分层设计**：从基础异常到具体业务异常的层级结构
- **标准化**：统一的错误响应格式和状态码
- **信息丰富**：包含足够的上下文信息
- **可追踪**：便于调试和问题定位

### 1.2 异常层级结构

```
ElevenBlogException (基类)
├── AgentException
│   ├── AgentNotFoundError
│   └── AgentExecutionError
│
├── LLMException
│   ├── LLMConnectionError
│   ├── LLMResponseError
│   └── LLMTimeoutError
│
├── RAGException
│   ├── DocumentProcessingError
│   ├── VectorDBError
│   └── EmbeddingError
│
├── ToolException
│   ├── ToolNotFoundError
│   └── ToolExecutionError
│
└── ConfigException
    ├── ConfigNotFoundError
    └── ConfigValidationError
```

## 2. 基础异常类

### 2.1 ElevenBlogException

**说明**：所有自定义异常的基类，提供统一的异常结构和方法。

**构造参数**：

- `message` (str): 错误信息
- `error_code` (int, 默认 500): HTTP 状态码
- `details` (Dict\[str, Any], 默认 {}): 详细上下文信息

**方法**：

- `to_dict()`: 将异常转换为字典格式，用于 API 响应

**使用示例**：

```python
from eleven_blog_tunner.core.exceptions import ElevenBlogException

# 抛出异常
raise ElevenBlogException(
    message="发生错误",
    error_code=500,
    details={"key": "value"}
)

# 捕获异常
try:
    # 执行操作
    pass
except ElevenBlogException as e:
    error_dict = e.to_dict()
    print(error_dict)
```

**错误响应格式**：

```json
{
  "error": {
    "message": "发生错误",
    "error_code": 500,
    "details": {
      "key": "value"
    }
  }
}
```

## 3. Agent 相关异常

### 3.1 AgentException

**说明**：Agent 相关异常的基类。

### 3.2 AgentNotFoundError

**说明**：当请求的 Agent 不存在时抛出。

**构造参数**：

- `agent_name` (str): 不存在的 Agent 名称

**错误码**：404

**使用示例**：

```python
from eleven_blog_tunner.core.exceptions import AgentNotFoundError

# 当 Agent 不存在时
if agent_name not in agent_registry:
    raise AgentNotFoundError(agent_name=agent_name)
```

**错误响应**：

```json
{
  "error": {
    "message": "Agent 'test_agent' not found",
    "error_code": 404,
    "details": {
      "agent_name": "test_agent"
    }
  }
}
```

### 3.3 AgentExecutionError

**说明**：当 Agent 执行失败时抛出。

**构造参数**：

- `agent_name` (str): 执行失败的 Agent 名称
- `error` (str): 错误原因

**错误码**：500

**使用示例**：

```python
from eleven_blog_tunner.core.exceptions import AgentExecutionError

try:
    result = await agent.execute(context)
except Exception as e:
    raise AgentExecutionError(
        agent_name=agent.name,
        error=str(e)
    )
```

**错误响应**：

```json
{
  "error": {
    "message": "Agent 'writer_agent' execution failed: LLM API error",
    "error_code": 500,
    "details": {
      "agent_name": "writer_agent",
      "error": "LLM API error"
    }
  }
}
```

## 4. LLM 相关异常

### 4.1 LLMException

**说明**：LLM 相关异常的基类。

### 4.2 LLMConnectionError

**说明**：当无法连接到 LLM 服务时抛出。

**构造参数**：

- `provider` (str): LLM 提供商名称
- `error` (str): 连接错误信息

**错误码**：503

**使用示例**：

```python
from eleven_blog_tunner.core.exceptions import LLMConnectionError

import httpx

try:
    response = httpx.post(api_url, json=data)
except httpx.ConnectError as e:
    raise LLMConnectionError(
        provider="openai",
        error=str(e)
    )
```

**错误响应**：

```json
{
  "error": {
    "message": "LLM connection failed (openai): Connection refused",
    "error_code": 503,
    "details": {
      "provider": "openai",
      "error": "Connection refused"
    }
  }
}
```

### 4.3 LLMResponseError

**说明**：当 LLM 返回错误响应时抛出。

**构造参数**：

- `provider` (str): LLM 提供商名称
- `error` (str): 响应错误信息

**错误码**：500

**使用示例**：

```python
from eleven_blog_tunner.core.exceptions import LLMResponseError

response = client.chat.completions.create(**params)
if response.error:
    raise LLMResponseError(
        provider="openai",
        error=response.error.message
    )
```

**错误响应**：

```json
{
  "error": {
    "message": "LLM response error (openai): Invalid API key",
    "error_code": 500,
    "details": {
      "provider": "openai",
      "error": "Invalid API key"
    }
  }
}
```

### 4.4 LLMTimeoutError

**说明**：当 LLM 请求超时时抛出。

**构造参数**：

- `provider` (str): LLM 提供商名称
- `timeout` (float): 超时时间（秒）

**错误码**：408

**使用示例**：

```python
from eleven_blog_tunner.core.exceptions import LLMTimeoutError

import asyncio

try:
    result = await asyncio.wait_for(llm.chat(messages), timeout=30.0)
except asyncio.TimeoutError:
    raise LLMTimeoutError(
        provider="openai",
        timeout=30.0
    )
```

**错误响应**：

```json
{
  "error": {
    "message": "LLM timeout (openai): 30.0s",
    "error_code": 408,
    "details": {
      "provider": "openai",
      "timeout": 30.0
    }
  }
}
```

## 8. 常见错误码说明

| 错误码 | 含义    | 适用场景                       |
| --- | ----- | -------------------------- |
| 400 | 错误请求  | 配置验证失败、文档格式错误              |
| 404 | 未找到   | Agent 未找到、工具未找到、配置未找到      |
| 408 | 请求超时  | LLM 超时                     |
| 500 | 服务器错误 | Agent 执行失败、工具执行失败、LLM 响应错误 |
| 503 | 服务不可用 | LLM 连接失败                   |

## 9. 最佳实践

### 9.1 异常抛出

- **具体异常优先**：尽量使用具体的异常类，而不是基础异常
- **提供详细信息**：在异常中包含足够的上下文信息
- **一致的错误信息**：错误信息应该清晰、准确，便于理解

**推荐**：

```python
# 好的做法
raise AgentNotFoundError(agent_name="test_agent")

# 不推荐
raise ElevenBlogException(
    message="Agent not found",
    error_code=404,
    details={"agent_name": "test_agent"}
)
```

### 9.2 异常捕获

- **分层捕获**：从具体到一般的捕获顺序
- **适当的处理**：根据异常类型采取不同的处理策略
- **记录日志**：捕获异常时记录详细的日志信息

**推荐**：

```python
try:
    result = await agent.execute(context)
except AgentNotFoundError as e:
    # 处理 Agent 未找到的情况
    logger.warning(f"Agent not found: {e.message}")
    return {"error": e.to_dict()}
except AgentExecutionError as e:
    # 处理 Agent 执行失败的情况
    logger.error(f"Agent execution failed: {e.message}")
    return {"error": e.to_dict()}
except Exception as e:
    # 处理其他未知错误
    logger.error(f"Unexpected error: {str(e)}")
    return {"error": {"message": "Internal server error"}}
```

### 9.3 API 响应

- **统一格式**：所有 API 错误响应使用统一的格式
- **适当的状态码**：根据异常类型返回对应的 HTTP 状态码
- **详细的错误信息**：返回足够的错误信息，便于客户端调试

**示例**：

```python
from fastapi import FastAPI, HTTPException
from eleven_blog_tunner.core.exceptions import ElevenBlogException

app = FastAPI()

@app.exception_handler(ElevenBlogException)
async def handle_eleven_blog_exception(request, exc):
    error_dict = exc.to_dict()
    raise HTTPException(
        status_code=exc.error_code,
        detail=error_dict["error"]
    )
```

## 10. 扩展异常类

如果需要添加新的异常类型，可以按照以下步骤：

1. **选择合适的父类**：根据异常类型选择合适的父类
2. **实现构造函数**：调用父类构造函数，提供必要的参数
3. **设置错误码**：根据异常类型设置合适的 HTTP 状态码
4. **添加到文档**：在本文档中添加新异常的说明

**示例**：

```python
from eleven_blog_tunner.core.exceptions import RAGException

class IndexingError(RAGException):
    """索引错误"""
    def __init__(self, error: str):
        super().__init__(
            message=f"Indexing failed: {error}",
            error_code=500,
            details={"error": error}
        )
```

## 11. 总结

ELEVEN Blog Tuner 的异常处理体系提供了：

- **分层的异常结构**：从基础到具体的异常层级

- **丰富的上下文信息**：包含详细的错误详情
- **清晰的错误码**：对应 HTTP 状态码，便于前端处理
- **灵活的扩展性**：易于添加新的异常类型

通过使用这套异常处理体系，可以：

- 提高代码的可读性和可维护性

- 便于调试和问题定位
